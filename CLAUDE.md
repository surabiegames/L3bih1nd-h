# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Backend/dashboard system for PERUMDA Tirtawening (Kota Bandung water utility), built on Next.js 16 with a Hono API layer and Prisma 7 + PostGIS. Domain: customer/meter management (`Pelanggan`, `Meter`), meter reading and billing workflows (`PembacaanMeter`, `Tagihan`, `TagihanLain`, `RiwayatPembayaran`), field/customer reporting (`LaporanHarianPetugas`, `LaporanMandiri`, `Pengaduan`), service-area geography (wilayah hierarchy + PostGIS geometry), org structure (`Divisi` → `Bagian` → `SubBagian`), and connection/disconnection/mutation tracking (`MutasiPelanggan`, `Pemutusan`). Many field/domain names are in Indonesian — keep new code consistent with that convention rather than translating to English.

## Commands

- `pnpm dev` — start Next.js dev server
- `pnpm db:bootstrap-admin` — create/reset the first `SUPER_ADMIN` (reads `BOOTSTRAP_ADMIN_EMAIL` — required, must match the Google account used to sign in — plus optional `_NAME`/`_USERNAME`/`_PASSWORD`; idempotent). **Required once on any fresh database**: the seed creates no `User` rows and there is no API path to the first admin (Credentials needs an existing user, Google sign-in rejects unknown emails, `POST /api/v1/users` needs an authenticated SUPER_ADMIN) — without this the whole system is locked out. Deliberately separate from `db:seed` (CSV import) so creating a privileged account is always an explicit act.
- `pnpm build` — production build
- `pnpm start` — run production build
- `pnpm lint` — ESLint (flat config: `eslint-config-next` core-web-vitals + typescript)
- `npx prisma generate` — regenerate the Prisma client after any schema change (see Prisma section below — this project is not using the default generated-client path)
- `npx prisma migrate dev` — create/apply a migration during development
- No test suite/runner is configured in this repo yet. Because of that, `tsc --noEmit` + `pnpm lint` are **not** sufficient verification for API changes: run `pnpm build && pnpm start`, log in, and exercise at least one authenticated geo endpoint — the two worst bugs found while building `server/**` (UntrustedHost, raw-SQL param binding) both passed typecheck and only surfaced on a real request.

Package manager is **pnpm** (`pnpm-lock.yaml`, `pnpm-workspace.yaml`); don't introduce npm/yarn lockfiles.

## Architecture

### Two server layers in one Next.js app

- `app/api/[[...route]]/route.ts` is a thin adapter only: it `handle()`s the Hono `app` exported from `server/app.ts` for `GET/POST/PUT/PATCH/DELETE`. **All API code lives in `server/**`, not under `app/`** (files inside `app/` get treated as routes/pages by the app router). Auth.js is wired into Hono via `@hono/auth-js` (`initAuthConfig`, `authHandler`, `verifyAuth`) rather than the Next.js-native Auth.js route handler. Add new endpoints as a module under `server/modules/<domain>/` and mount it in `server/app.ts` — never as a separate `app/api/**/route.ts` file.
- **See `server/README.md`** for the endpoint map, response/error envelope contract, RBAC groups, the business rules the API enforces that the DB can't, and three non-obvious traps that already bit once (raw-SQL `Prisma.raw()`, `instanceof` error checks, `trustHost`). Read it before touching `server/**`.
- Two conventions that are easy to break and cost real debugging time:
  - **Never use `Prisma.raw()`/`Prisma.join()` or `instanceof` (for Prisma/Zod errors) inside `server/**`.** The generated Prisma client can be loaded as more than one module instance (globalThis-cached client + HMR-reloaded module), so `instanceof Sql` / `instanceof PrismaClientKnownRequestError` silently return false. Verified concretely: raw identifiers got bound as parameters (`FROM $3` → `syntax error at or near "$3"`), and the entire P2002/P2003 → 409/400 mapping was dead code returning 500. Use `$queryRawUnsafe`/`$executeRawUnsafe` (identifiers from the `GEO` whitelist in `server/lib/spatial.ts`, values as positional params) and structural error checks (`server/lib/errors.ts`).
  - **Use `validate()` from `server/lib/validate.ts`, not `zValidator` directly** — the raw validator returns its own 400 with a ZodError dump and never reaches `app.onError`, bypassing the error envelope.
- `auth.config.ts` sets `trustHost: true`; without it `pnpm build && pnpm start` returns 401 `UntrustedHost` on every authenticated endpoint while `next dev` looks fine. Set `AUTH_URL` to the canonical origin in production.
- **Mobile (Flutter) clients use Bearer tokens, not cookies**: `POST /api/mobile/auth/login|google` (public, rate-limited, uniform failure message) issues a 7-day Auth.js-JWE token with a *different salt* than the web session cookie; the `/api/v1` blanket is `verifyAuthFleksibel()` (`server/lib/mobile-token.ts`) which accepts cookie **or** Bearer and still defaults to 401 — don't replace it with plain `verifyAuth()` or Bearer auth silently dies. CORS is registered *before* the routes in `server/app.ts` (Hono middleware only applies to routes registered after it); origins come from `CORS_ORIGINS` env. See "API untuk aplikasi mobile (Flutter)" in `server/README.md`. **`FLUTTER.md`** is the self-contained client-side spec (API contract, auth flow, endpoint catalog, Dart conventions) meant to be copied into / read by the separate Flutter project — keep it in sync when the API contract changes.
- `lib/prisma.ts` is the single PrismaClient singleton (PrismaPg adapter + globalThis cache) — `auth.ts` and all of `server/**` import from it. Don't construct a second `PrismaClient` in app code.
- `proxy.ts` exports `proxy` (aliased `auth` from `next-auth`) built from `authConfig` alone, used for edge-safe route protection (see `authConfig.callbacks.authorized`, which currently gates any path under `/dashboard`). It matches all paths except `/api`, `_next/static`, `_next/image`, `favicon.ico`.

### Auth split: `auth.config.ts` vs `auth.ts`

- `auth.config.ts` holds the **edge-safe** config: providers (Google + a stubbed Credentials provider), pages, the `authorized` callback, `session: { strategy: "jwt" }`. It deliberately does **not** import Prisma (`authorize` here always returns `null`) so it can run in edge/middleware contexts.
- `auth.ts` holds the **Node-only** config: builds the real Prisma-backed `PrismaAdapter`, replaces the stub Credentials provider with a real one that checks `username`/`passwordHash` via bcrypt, and adds `jwt`/`session`/`signIn` callbacks that propagate `role`, `divisiKode`, `subBagianKode` onto the token/session. Google sign-in is restricted to emails that already exist as a `User` row (no self-service signup via Google).
- When touching auth, keep this split intact: anything that needs Prisma belongs in `auth.ts`, not `auth.config.ts`.
- Session/JWT/User types are augmented in `types/next-auth.d.ts` to add `role`, `divisiKode`, `subBagianKode`.

### Prisma: multi-file schema, 6 domain files

- Schema is split into 6 files under `prisma/` (see `prisma/README.md` for the full model map and the 2026-07-14/2026-07-15 change notes): `schema.prisma` (generator/datasource only — required to be minimal for multi-file mode), `auth.prisma` (User/Account/Session/TwoFactor + Role/UserStatus), `organisasi.prisma` (Divisi→Bagian→SubBagian, `Pencatat` bridge, `TargetKinerja`), `pelanggan.prisma` (wilayah hierarchy, `Dma`, Kecamatan/Kelurahan, tarif, `GolonganBesar`/`ObjekBayar`, `Pelanggan`), `tagihan.prisma` (`Meter`, `PembacaanMeter`, `Tagihan`, `TagihanLain`, `RiwayatPembayaran`), `operasional.prisma` (`Pengaduan`, `LaporanHarianPetugas`, `LaporanMandiri`, `MutasiPelanggan`, `Pemutusan`, `PotensiPelanggan`, `AuditLog`, `Konfigurasi`). `introspected.prisma` (PostGIS's own `spatial_ref_sys` table) is intentionally kept separate from these 6 as it's infrastructure, not domain.
- `prisma.config.ts` points Prisma at `schema: "prisma"` (the directory) and `migrations.path: "prisma/migrations"`.
- Generator `output` in `schema.prisma` is `../app/generated/prisma`, matching what app code actually imports (`@/app/generated/prisma`, `@/app/generated/prisma/client`, gitignored). Prisma 7 no longer allows a `url` in the `datasource` block (`prisma validate` errors with P1012 if you add one) — the connection string only lives in `prisma.config.ts` (CLI) and is passed explicitly as a driver adapter to `PrismaClient` (runtime).
- The `db` datasource uses the `postgis` extension (`previewFeatures = ["postgresqlExtensions"]`). Geometry columns are declared as `Unsupported("geometry(Point, 4326)")` for point columns or `Unsupported("geometry(MultiPolygon, 4326)")` for area/boundary columns, each with an explicit `Gist` index — raw SQL (`$queryRaw`/`$executeRaw`) is required for any actual geometry read/write/query, Prisma's query builder can't touch `Unsupported` fields directly. See the PostGIS usage guide at the end of `prisma/README.md` for concrete `ST_*` query patterns.
- Prisma client is created with the `@prisma/adapter-pg` driver adapter (`PrismaPg`, using `DATABASE_URL`), not Prisma's built-in connection handling — instantiate new PrismaClients the same way (`new PrismaClient({ adapter })`), and reuse the `globalThis`-cached singleton pattern from `auth.ts` to avoid connection storms in dev.
- **Always `import { PrismaClient } from "@/app/generated/prisma"` (the package root), never from `"@/app/generated/prisma/client"`.** Verified concretely: importing via the `/client` subpath (a bare `export * from "./index"` re-export) makes TypeScript fail to infer the `ClientOptions` generic, silently dropping 33 of 37 model delegate getters from the type (`Property 'x' does not exist on type 'PrismaClient<never, ...>'`) — only `user`/`account`/`session`/`spatial_ref_sys` happened to still resolve. Importing from the root package doesn't have this problem at all. Runtime behavior is identical either way (both subpaths point at the same `index.js`); this is purely a type-resolution footgun.
- `app/generated/prisma` also needs `@prisma/client-runtime-utils` as a **direct** dependency of this project (already added to `package.json`), even though it's only ever `require()`'d by Prisma's own generated runtime code. Reason: the generated client lives inside the app source tree, not inside `node_modules`, so pnpm's strict per-package isolation never hoists this transitive-of-`@prisma/client` dependency to somewhere Node's module resolution can find it when walking up from `app/generated/prisma/runtime/`. Without it, `next build` fails with `Module not found: Can't resolve '@prisma/client-runtime-utils'`.
- Keep TypeScript pinned to the `^5` range. TS 6/7 are out but Next.js 16.2.10's build tooling doesn't recognize them yet (`next build` fails with "It looks like you're trying to use TypeScript but do not have the required package(s) installed", then crashes trying to auto-install its own copy) — this was tested directly, not assumed.
- Every domain model has an explicit snake_case `@@map(...)` (matching the pattern Auth.js tables already required) so Postgres table names don't depend on case-sensitive quoted identifiers.
- `prisma/README.md` documents non-obvious data-modeling decisions worth reading before touching related models: e.g. `Meter.isAktif` handles duplicate `nomorMeter` history instead of a unique constraint, `nprs` is intentionally not `@unique` (duplicate physical locations, different customers), `Pemutusan.pelangganId` and `LaporanHarianPetugas.nomorLangganan` are optional because field/survey data references customers outside the main CSV import window, and Kelurahan `area` columns must be `MultiPolygon` (some real boundaries have multiple disjoint parts).
- `RiwayatPembayaran` is a single payment ledger shared by both `Tagihan` (periodic water bills) and `TagihanLain` (one-off non-water charges: new connections, name transfer, meter replacement, fines) — exactly one of `tagihanId`/`tagihanLainId` should be set per row; Prisma has no native polymorphic relation to enforce that, so it must be validated at the application layer.
- **Riwayat migrasi SINKRON penuh dengan DB sejak 2026-07-19** (8 migrasi, terakhir `20260719120000_langganan_warga_dan_pengaduan_lengkap` — baseline dari perubahan yang sempat masuk via `db push`; ditandai `migrate resolve --applied` di DB live dan diverifikasi dengan replay `migrate deploy` penuh di DB segar). **Aturan ke depan: perubahan skema = migrasi (`prisma migrate dev`), BUKAN `db push`** — `db push` yang membuat riwayat tertinggal dan `migrate dev` menawarkan reset destruktif. `migrate dev`/`migrate diff --from-migrations` butuh **shadow DB** karena `DATABASE_URL` menunjuk Neon (tak boleh CREATE DATABASE): set `SHADOW_DATABASE_URL` ke database sekali-pakai di Postgres lokal docker-compose (mis. `postgresql://dev:dev@127.0.0.1:5432/shadow_uji` — buat dulu via `docker exec pg-dev psql -U dev -c 'CREATE DATABASE shadow_uji;'`), sudah dibaca `prisma.config.ts`. `prisma migrate reset` terhadap `DATABASE_URL` produksi tetap langkah destruktif yang hanya boleh dipicu manusia secara sadar.
  - **Update 2026-07-20**: migrasi terakhir kini `20260720222024_idempotensi_pengaduan` (kolom `Pengaduan.clientRequestId` unik untuk idempotensi pengaduan publik). **JEBAKAN yang sudah kena**: `prisma migrate dev` terhadap Neon melaporkan *drift* palsu `[+] Added extensions: postgis` (ekstensi ADA di DB tapi tak terlacak di riwayat migrasi) lalu menawarkan **reset destruktif** — JANGAN dituruti. Jalur aman untuk perubahan skema ADITIF ke DB live: (1) `prisma migrate diff --from-config-datasource --to-schema prisma --script` untuk melihat SQL persisnya (introspeksi DB live vs datamodel, tanpa noise postgis), (2) tulis file migrasi manual di `prisma/migrations/<ts>_<nama>/migration.sql`, (3) terapkan dengan **`prisma migrate deploy`** (hanya menjalankan migrasi pending, tanpa shadow/drift-check), (4) `prisma generate`. `migrate dev` tetap boleh untuk perubahan lokal di DB segar, tapi bukan terhadap Neon berisi data.
- Relation pattern for CSV-sourced models that may reference a `Pelanggan` not yet imported (`LaporanHarianPetugas`, `Pemutusan`): an **optional** `@relation` FK (so it's `include`-able from the frontend once resolved) *plus* a raw text snapshot field (`nomorLangganan`, `namaPelanggan`, `alamatPelanggan`) that's always populated at import time so orphaned rows never lose their identity. Follow this same two-part pattern (relation + snapshot) for any new model that ingests legacy CSV data keyed by `nomorLangganan`.
- ETL philosophy (see `prisma/README.md` "Filosofi ETL"): raw CSVs in `prisma/data/` are immutable source archives — never hand-edit them. All normalization (zero-padding `nolg`, trimming stray whitespace, Excel-serial-date parsing, code→enum mapping) belongs in the seed/import script itself, so it's reusable for every future monthly export and testable against the edge cases already found in this data.

### Org/permissions model

`Role` (enum: `SUPER_ADMIN`, `DIREKSI`, `SENIOR_MANAGER`, `MANAGER`, `SUPERVISOR`, `STAFF`, `USER`) is a flat enum on `User`, cross-referenced against the `Divisi → Bagian → SubBagian` org hierarchy (`Bagian.levelKepala` records the expected head role per bagian, e.g. `SENIOR_MANAGER` in PELAYANAN vs `MANAGER` in TEHNIK/UMUM — this is documentation, not an enforced constraint). `User` carries both relational FKs (`divisiId`/`bagianId`/`subBagianId`) and denormalized `divisiKode`/`subBagianKode` used directly in the JWT/session — keep both in sync if you change org assignment logic. `Pencatat` is a separate bridge model mapping field-report "petugas" names (free text in source CSVs) to real `User` accounts, since field data historically used nicknames rather than system usernames.

### Frontend structure

**See `FRONTEND.md`** for the page/component pattern — read it before adding any page or UI component. `/login` (`app/(auth)/login/` + `features/auth/`) is the worked reference example for every rule in it. The short version:

- `app/**` holds route concerns only (metadata, params, session guard, redirect) and composes from `features/<domain>/` — which mirrors `server/modules/<domain>/` (API for a domain in `server/modules/pelanggan/` → its UI in `features/pelanggan/`). Long JSX and domain logic don't belong in `page.tsx`.
- Server components by default; `"use client"` goes on the leaf that actually needs interactivity, never on `page.tsx`/`layout.tsx`.
- Route groups (`(auth)`, later `(dashboard)`) group pages by shared layout without adding a URL segment.
- Colors only via semantic tokens (`bg-background`, `text-muted-foreground`, …) — raw `bg-white`/`text-gray-500`/hex silently break dark mode, since only the tokens have `.dark` counterparts.
- Dark mode is class-based: `globals.css` uses `@custom-variant dark (&:is(.dark *))`, so `ThemeProvider` must stay `attribute="class"` and `<html>` must keep `suppressHydrationWarning`. For theme-dependent UI use the `dark:` variant (CSS decides), not a `useState(mounted)` + `useEffect` dance — that flickers and trips `react-hooks/set-state-in-effect`.
- shadcn style is `radix-nova`/`mist`; add components with `npx shadcn@latest add <name>`, never hand-edit `components/ui/**` (it's generated and intentionally not lint-clean — lint your own files, not that folder).
- Forms: server action + `useActionState` by default (`features/auth/components/forgot-password-form.tsx`); a client handler only when submit must go through someone else's endpoint (`credentials-form.tsx` → Auth.js). **Every exported async function in a `"use server"` file is a browser-callable RPC endpoint** — only actions belong in `features/*/actions/`; internal helpers live in `features/*/lib/` with `import "server-only"` at the top.

### Public (unauthenticated) surface

`/cek-tagihan`, `/lapor-meter`, `/pengaduan` (`app/(public)/` + `features/publik/`) are served by **`/api/public/*` — a router deliberately separate from `/api/v1`**. Full rationale in `FRONTEND.md`; the load-bearing rules:

- **Never move a public endpoint into `/api/v1` and poke a hole in its blanket `verifyAuth()`.** That blanket is the safety net that keeps an internal endpoint closed when someone forgets `requireRole()`.
- **Customer-data endpoints require `nomorLangganan` + `nama`, not the number alone.** Measured, not assumed: `nomorLangganan` is *sequential* (`00000100119`, `00000200509`, …) so a loop would harvest all 22,534 customers; phone can't be the second factor (only **16.3%** have `notelp`, some `"0000"`); `nama` is viable (**1 of 22,534** unusable). This is anti-harvesting, not authentication.
- **Identical failure message for every verification failure** (unknown number vs wrong name) — differentiating turns the endpoint into a "which numbers exist?" oracle. Same reason `/pengaduan` stores `nomorLangganan` as free text and never verifies it.
- Public meter reports are always `MENUNGGU` and the period is server-decided; staff verification is what turns them into official readings.
- Uploads are validated by **magic bytes**, not client-supplied `file.type`; server picks the filename/extension (`server/lib/storage.ts`).
- `server/lib/rate-limit.ts` is **in-process** (lost on restart, per-instance) and trusts `x-forwarded-for` — a prod reverse proxy must overwrite that header or every limit is bypassable. Multi-instance ⇒ move to Redis.
- **Uploads go to Cloudinary** (`server/lib/storage.ts`), signed upload — never an unsigned preset (that would let anyone reading the browser JS burn our quota). Falls back to `./.uploads` in dev, throws in production. **The current credentials are rejected (401)** — diagnosed conclusively: Cloudinary's reported `String to sign` matches ours exactly and `sha1(string + our secret)` equals what we sent, so the algorithm is right and only `CLOUDINARY_API_SECRET` can be wrong. Don't "fix" the signing code; fix the credential.

### Dashboard

`app/(dashboard)/` + `features/dashboard/`. See `FRONTEND.md`. Non-obvious parts:

- Server components read data **directly via `features/dashboard/lib/queries.ts`**, not `fetch("/api/v1/dashboard")` — same server, so HTTP to itself is a wasted hop plus manual cookie forwarding plus an absolute-URL problem. The `/api/v1/dashboard/*` endpoints exist for *external* callers. (Public pages are the opposite: they must fetch, because rate limiting keys on the user's IP.)
- Aggregate in the DB (`count`/`aggregate`/`groupBy`); never `findMany().length` — pelanggan and tagihan are >22k rows each.
- Reference period = **latest period that has data**, not the current month; monthly closing lags, and using the current month makes the dashboard look broken when it's just early in the month.
- Money in stat cards uses `formatRupiahRingkas()` — full values truncate to `Rp 3.012.3…` and become unreadable (observed, not hypothetical). Full value stays in `title` and in tables.
- Charts need ≥2 points; Recharts renders an empty box with one, which reads as a broken component. `TrenTagihan` degrades to plain numbers + an explanation — the seeded data currently *is* single-period.

### Auth flows

`/login` (Google + credentials), `/forgot-password`, `/reset-password` exist; there is deliberately **no signup** (admin-provisioned via `POST /api/v1/users`; `callbacks.signIn` rejects unregistered Google emails). Details and the security rules behind them are in `FRONTEND.md` — read before touching `features/auth/**`. Load-bearing points:

- **Passwords: argon2id via `@node-rs/argon2`, and `lib/password.ts` is the only place that hashes/verifies** — never call the hashing library directly elsewhere. Auth.js provides no password hashing at all (verified: zero bcrypt/argon2/scrypt in `@auth/core`/`next-auth`; deps are just jose/oauth4webapi/hkdf/preact), so the choice is ours and there is no "Auth.js standard" to match. `@node-rs/argon2`'s defaults (`m=19456,t=2,p=1`) already equal OWASP's recommendation — don't override. Params live inside the hash string, so raising them later won't invalidate old hashes.
  - It's a **native module**: verified working on Windows dev; `pnpm install` on the prod host must fetch the matching prebuild (usually linux-x64-gnu).
  - `DUMMY_PASSWORD_HASH` must stay a **real argon2id hash** — measured: a malformed string is rejected by the parser in ~0.1ms vs ~25ms for a real hash (389× faster), which reintroduces exactly the timing leak it exists to prevent.
  - `lib/password.ts` imports `server-only`, so any plain-Node script using it (e.g. `prisma/seed/bootstrap-admin.ts`) must run under `tsx --conditions=react-server` — already wired into `pnpm db:bootstrap-admin`.
- Reset tokens reuse the idle `VerificationToken` table (no migration) and are stored **SHA-256 hashed**, single-use, 1-hour TTL — `features/auth/lib/password-reset.ts`.
- Credentials `authorize()` matches **email OR username** (accounts created for Google login have `username = null`, so username-only lookup made them unable to sign in at all), rejects non-`ACTIVE` users, and always runs `bcrypt.compare` against a dummy hash so response time doesn't reveal which emails exist. The Google `signIn` callback checks `status` too — checking only one path leaves the other door open.
- Never differentiate error messages by cause (unknown email vs wrong password vs suspended); `/forgot-password` always returns the same success message. That's user-enumeration defense, not vagueness.
- **Email is not configured yet**: `features/auth/lib/mailer.ts` uses Resend if `RESEND_API_KEY` is set, otherwise logs the mail to the server console in dev and **throws in production** (silently not sending a reset link is worse than a visible failure).
- Sessions are JWT and cannot be revoked server-side, so a password reset does **not** kill sessions on other devices. Changing that means moving to database sessions — a deliberate decision, not a quiet edit.

### Path alias

`@/*` resolves to the repo root (`tsconfig.json`), e.g. `@/auth.config`, `@/app/generated/prisma`, `@/features/auth/components/login-card`.
